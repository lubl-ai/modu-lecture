## 섹션 1. 문서 개요

### 1.1 목적 (Purpose)

본 문서는 금태 프리미엄 해외 수출 서비스(GEUMTAE Export)의 소프트웨어 요구사항을 정의한다. MVP 단계에서 개발자, QA, 외부 통합 파트너가 시스템 구현과 검증의 기준으로 사용한다.

### 1.2 범위 (Scope)

**시스템명**: GEUMTAE Export 운영 관리 시스템

**포함 범위**
- 발주~납품~정산 이력 관리 (Order, Shipment, QualityReport, Claim)
- IoT 온도 센서 데이터 수집 및 이탈 알림
- 물류 실시간 추적 링크 생성 및 바이어 자동 발송
- 통관 서류 체크리스트 관리 및 EDI 연동
- 일본어 CS 창구 (LINE Messaging API 연동)
- 월간 납품 실적 리포트 자동 생성

**제외 범위**
- B2C 플랫폼 및 소비자 직판
- 가공식품 제조 관리
- 일본 현지 법인 운영 시스템
- 자사 물류 차량 관제

### 1.3 의도된 독자 (Intended Audience)

| 역할 | 사용 목적 |
|------|-----------|
| 백엔드 개발자 | 기능 구현 기준, API 인터페이스 명세 |
| QA 엔지니어 | 테스트 케이스 작성 기준, 수용 기준 |

| 물류 파트너 | IoT 연동 요건 파악 |
| 통관 에이전트 | 서류 연동 인터페이스 확인 |

## 섹션 2. 시스템 개요

### 2.1 시스템 설명 (System Description)

GEUMTAE Export 운영 관리 시스템은 한국산 금태의 B2B 수출 운영을 지원하는 소규모 팀 대상 백오피스 시스템이다. 발주 접수부터 납품 완료, 서류 관리, 클레임 처리까지 전체 운영 워크플로우를 단일 시스템으로 통합한다.

운영 인원: CS 담당자 1인, PO 1인, 물류 파트너(외부)
기술 환경: AWS ECS (컨테이너), PostgreSQL, AWS IoT Core (MQTT), LINE Messaging API, 물류 추적 API

### 2.2 Context Diagram (텍스트 표현)

```
[외부 엔티티]                    [GEUMTAE Export 시스템]                  [외부 엔티티]
                                  
바이어 (일본 레스토랑)  ──발주 요청──>  [발주 관리 모듈]  ──출하 지시──>  양식장 공급자
                      <──추적 링크──   [물류 추적 모듈]  <──추적 이벤트── DHL/FedEx/EMS API
                      <──서류 첨부──   [서류 관리 모듈]  <──신고필증──   UNI-PASS EDI
                                      [IoT 모니터링]   <──온도 데이터── IoT 센서 (MQTT)
CS 담당자           ──메시지──>        [CS 모듈]        ──알림──>       LINE Messaging API
PO                 ──조회──>          [리포트 모듈]
물류 파트너          ──배송 완료──>     [납품 확정 모듈]
```

### 2.3 운영 환경 (Operating Environment)

| 항목 | 사양 |
|------|------|
| 클라우드 | AWS (1순위) / GCP (2순위) |
| 컨테이너 | ECS Fargate 또는 Cloud Run |
| 데이터베이스 | PostgreSQL 15+ (RDS 또는 Cloud SQL) |
| IoT 브로커 | AWS IoT Core (MQTT 3.1.1) |
| 런타임 | Node.js 20 LTS 또는 Python 3.11+ |
| 가용성 목표 | 월 99.9% 업타임 |
| 예산 제약 | 월 클라우드 비용 $200 이하 (MVP 단계) |

### 2.4 설계 및 구현 제약 (Constraints)

- 백엔드 개발자 1인 운영 → 모노리식 또는 모듈형 모노리스 구조 채택
- 외부 서비스 장애 시 운영 차질 방지를 위한 폴백 처리 필수
- DB 및 인터페이스 UTF-8 인코딩 필수 (일본어 처리)
- 바이어 대상 인터페이스는 앱 설치 불필요 (LINE/KakaoTalk 기반)
- 수출신고필증 5년 보관 (법정 요건)

## 섹션 3. 이해관계자 요구사항 (StRS)

### StRS-PO: 제품 오너 (사업 대표)

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| StRS-PO-01 | 납품 건별 SLA 준수 여부를 실시간으로 조회할 수 있어야 한다 | 필수 |
| StRS-PO-02 | 월간 납품 실적(출하량, 매출, SLA 준수율, 반품률)이 자동으로 집계되어야 한다 | 필수 |
| StRS-PO-03 | 바이어별 재주문 현황과 계약 만료 일정을 한눈에 파악할 수 있어야 한다 | 권고 |

### StRS-CS: 일본어 전담 CS 담당자

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| StRS-CS-01 | 바이어 문의가 LINE을 통해 수신되면 시스템에 자동 기록되어야 한다 | 필수 |
| StRS-CS-02 | 발주 확정 시 바이어에게 추적 링크가 자동으로 발송되어야 한다 (수동 개입 없이) | 필수 |
| StRS-CS-03 | 클레임 접수 후 처리 기한(48시간) 초과 전에 CS 담당자에게 알림이 발송되어야 한다 | 필수 |

### StRS-LOG: 물류 파트너

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| StRS-LOG-01 | IoT 온도 데이터를 MQTT로 시스템에 전송할 수 있어야 한다 | 필수 |
| StRS-LOG-02 | 온도 이탈(5°C 초과) 발생 시 물류 파트너와 CS 담당자에게 동시 알림이 가야 한다 | 필수 |

### StRS-CUS: 통관 에이전트

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| StRS-CUS-01 | 납품 건별 필요 서류 체크리스트를 시스템에서 확인할 수 있어야 한다 | 필수 |
| StRS-CUS-02 | 수출신고필증 수신 후 시스템에 자동으로 연결되어야 한다 | 권고 |

### StRS-BUY: 바이어 (일본 레스토랑)

| ID | 요구사항 | 우선순위 |
|----|---------|---------|
| StRS-BUY-01 | 발주 후 물류 현황을 별도 앱 설치 없이 추적할 수 있어야 한다 | 필수 |
| StRS-BUY-02 | 납품 서류(HACCP 인증서, 위생증명서)가 납품과 함께 일본어로 제공되어야 한다 | 필수 |
| StRS-BUY-03 | 클레임 접수 후 48시간 이내에 처리 결과를 통보받아야 한다 | 필수 |

## 섹션 4. 시스템 요구사항 (SyRS)

### 4.1 기능 요구사항 (SRS-F)

#### SRS-F-001: HACCP 인증 문서 관리

| 항목 | 내용 |
|------|------|
| 설명 | 납품별 HACCP 로트번호와 인증서 사본을 시스템에 등록하고, 납품 확정 시 바이어에게 자동 첨부 |
| 수용 기준 | 납품 건 100%에 HACCP 로트번호 및 인증서 링크가 기록됨. 미기록 시 납품 확정 불가 |
| CQRS | Command (문서 등록), Query (서류 조회) |

#### SRS-F-002: 위생증명서 및 원산지증명서 생성

| 항목 | 내용 |
|------|------|
| 설명 | 납품 건 기준으로 위생증명서, 원산지증명서 초안을 자동 생성하고, 통관 에이전트 검토 후 확정 |
| 수용 기준 | 서류 생성 소요 시간 발주 확정 후 24시간 이내. 일본어 필드 UTF-8 정상 출력 확인 |
| CQRS | Command (서류 생성), Query (서류 상태 조회) |

#### SRS-F-003: 납기 SLA 계약 이행 기록

| 항목 | 내용 |
|------|------|
| 설명 | 납품 건별 약정 도착일을 등록하고, 실제 도착일과 비교하여 SLA 준수 여부를 자동 판정 |
| 수용 기준 | 실제 도착일 등록 시 SLA 준수 여부(준수/위반)가 1분 이내 자동 산정됨. 위반 시 PO에게 알림 발송 |
| CQRS | Command (도착일 등록), Query (SLA 현황 조회) |

#### SRS-F-004: 지연 보상 산정 자동화

| 항목 | 내용 |
|------|------|
| 설명 | SLA 위반 납품 건에 대해 계약 조건(지연일수 × 단가)에 따라 보상액을 자동 계산 |
| 수용 기준 | SLA 위반 등록 후 10분 이내에 보상액 초안이 생성됨. PO 최종 승인 후 확정 |
| CQRS | Command (보상 확정), Query (보상 내역 조회) |

#### SRS-F-005: IoT 온도 데이터 수집

| 항목 | 내용 |
|------|------|
| 설명 | MQTT를 통해 IoT 센서로부터 온도, 습도, GPS 위치 데이터를 수집하고 납품 ID와 매핑하여 저장 |
| 수용 기준 | 센서 데이터 수신 후 DB 저장까지 5초 이내. 납품 건 100%에 온도 로그 기록 |
| CQRS | Command (센서 데이터 저장) |

#### SRS-F-006: 온도 이탈 감지 및 알림

| 항목 | 내용 |
|------|------|
| 설명 | 온도가 5°C를 초과하면 즉시 이탈 이벤트를 생성하고, CS 담당자와 물류 파트너에게 LINE/SMS 알림 발송 |
| 수용 기준 | 이탈 감지 후 알림 발송까지 2분 이내. 월간 온도 이탈 발생률 1% 미만 유지 기록 |
| CQRS | Command (이탈 이벤트 기록), Query (이탈 이력 조회) |

#### SRS-F-007: 물류 추적 링크 자동 생성 및 발송

| 항목 | 내용 |
|------|------|
| 설명 | 발주 확정 시 물류 API(DHL/FedEx/EMS)를 호출하여 추적 URL을 생성하고 바이어 LINE 채널로 자동 발송 |
| 수용 기준 | 발주 확정 후 30분 이내 추적 링크 바이어 수신 완료. API 장애 시 수동 링크 입력 폴백 UI 제공 |
| CQRS | Command (링크 생성·발송), Query (추적 상태 조회) |

#### SRS-F-008: 물류 상태 갱신

| 항목 | 내용 |
|------|------|
| 설명 | 물류 API로부터 상태(출발/통관/배송중/도착 예정)를 폴링 또는 웹훅으로 수신하고 DB에 반영 |
| 수용 기준 | 상태 변경 발생 후 30분 이내 DB 갱신. 갱신 실패 시 3회 재시도 후 PO 알림 |
| CQRS | Command (상태 갱신) |

#### SRS-F-009: 통관 서류 체크리스트 관리

| 항목 | 내용 |
|------|------|
| 설명 | 납품 건별 필요 서류 목록(위생증명서, 원산지증명서, 수출신고필증 등)을 체크리스트로 관리하고 미완료 항목 추적 |
| 수용 기준 | 미완료 서류 항목이 있으면 납품 확정 버튼 비활성화. 완료 기록 타임스탬프 저장 |
| CQRS | Command (체크 완료 기록), Query (서류 현황 조회) |

#### SRS-F-010: 통관 EDI 연동 (UNI-PASS)

| 항목 | 내용 |
|------|------|
| 설명 | 한국 관세청 UNI-PASS EDI를 통해 수출신고를 자동 제출하고 수출신고필증을 수신하여 DB에 저장 |
| 수용 기준 | EDI 제출 후 수신 결과(성공/실패) 30분 이내 반영. 실패 시 에러 코드 및 원인 기록 |
| CQRS | Command (신고 제출), Query (신고 상태 조회) |

#### SRS-F-011: LINE 메시지 수신 및 기록

| 항목 | 내용 |
|------|------|
| 설명 | LINE Messaging API 웹훅을 통해 바이어 메시지를 수신하고 CS 화면에 표시, 대화 이력 DB 저장 |
| 수용 기준 | 메시지 수신 후 CS 화면 표시까지 1분 이내. 대화 이력 30일 이상 보관 |
| CQRS | Command (메시지 저장), Query (대화 이력 조회) |

#### SRS-F-012: CS 응답 SLA 타이머 관리

| 항목 | 내용 |
|------|------|
| 설명 | 바이어 문의 수신 시 24시간 타이머 시작. 미응답 시 22시간 시점에 CS 담당자에게 알림 발송 |
| 수용 기준 | 타이머 기산 시점은 메시지 수신 타임스탬프 기준. 알림 발송 성공률 99% 이상 |
| CQRS | Command (타이머 시작·초기화), Query (미응답 건 목록) |

#### SRS-F-013: 클레임 접수 및 처리 워크플로우

| 항목 | 내용 |
|------|------|
| 설명 | 선도 불량 클레임을 접수하고 상태(접수→검토→승인→처리완료)를 추적. 처리 기한 48시간 |
| 수용 기준 | 접수 후 46시간 시점 CS 담당자 알림. 처리완료 기록 후 바이어에게 결과 자동 통보 |
| CQRS | Command (클레임 접수·상태 변경), Query (클레임 이력 조회) |

#### SRS-F-014: 샘플 발송 관리 및 피드백 수집

| 항목 | 내용 |
|------|------|
| 설명 | 신규 바이어 대상 샘플 발송 건을 등록하고, 발송 후 14일 이내 피드백 수집 알림 자동 발송 |

### 4.3 인터페이스 요구사항

#### 외부 시스템 연동

| 시스템 | 연동 방식 | 데이터 포맷 | 폴백 처리 |
|--------|----------|------------|----------|
| DHL/FedEx/EMS 추적 API | REST API (HTTPS) | JSON | API 장애 시 수동 링크 입력 UI 제공 |
| UNI-PASS EDI (관세청) | EDI (UN/EDIFACT 또는 XML) | XML/EDI 표준 | 실패 시 수동 서류 업로드 경로 유지 |
| LINE Messaging API | HTTPS 웹훅 + REST | JSON | 웹훅 수신 실패 시 폴링 방식으로 전환 |
| AWS IoT Core (MQTT) | MQTT 3.1.1 over TLS | Binary/JSON | 연결 끊김 시 10분 간격 재연결 시도 |
| 환율 API (한국수출입은행) | REST API | JSON | 장애 시 전일 환율 캐시값 사용 |

#### IoT 센서 인터페이스

- 프로토콜: MQTT 3.1.1 over TLS (포트 8883)
- 토픽 구조: `geumtae/shipment/{shipmentId}/sensor`
- 페이로드: `{ "temp": float, "humidity": float, "lat": float, "lon": float, "ts": ISO8601 }`
- QoS: Level 1 (At least once)
- 센서 보고 주기: 15분 간격 (이탈 감지 시 즉시 보고)
