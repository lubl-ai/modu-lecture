## 1. PRD vs SRS 차이

| 항목 | PRD (Product Requirements Document) | SRS (Software Requirements Specification) |
|------|--------------------------------------|-------------------------------------------|
| 목적 | 무엇을 만들 것인지 정의 (비즈니스 관점) | 어떻게 동작해야 하는지 명세 (기술 관점) |
| 대상 독자 | 사업 오너, PM, 디자이너, 이해관계자 | 개발자, QA, 시스템 설계자 |
| 수준 | 기능 의도 + 비즈니스 목표 | 측정 가능한 동작 기준 + 인터페이스 명세 |
| 검증 기준 | "바이어가 만족하는가?" | "시스템이 조건 X를 충족하는가?" |
| 언어 | 자연어, 비즈니스 용어 | 형식적·구조화된 요구사항 표현 |
| 우선순위 | MoSCoW, OS 점수 등 | 필수(Mandatory) / 권고(Recommended) 구분 |
| 추적성 | 비즈니스 목표 ↔ 기능 | 요구사항 ID ↔ 구현 ↔ 테스트 |
| 변경 관리 | 비즈니스 의사결정 중심 | 기술 변경 영향 분석 포함 |

**핵심 차이**: PRD는 "고객이 원하는 것"을 설명하고, SRS는 "시스템이 해야 할 것"을 명세한다. PRD에서 "납기 준수율 95%"라는 목표가 SRS에서는 "SRS-F-003: 시스템은 발주 확정 후 30분 이내에 추적 링크를 생성해야 한다"와 같이 구체적 동작으로 분해된다.

## 2. ISO/IEC/IEEE 29148:2018 SRS 표준 구조

ISO/IEC/IEEE 29148:2018은 시스템 및 소프트웨어 요구사항 명세의 국제 표준으로, SRS를 7개 주요 섹션으로 구성한다.

### 섹션 1: 소개 (Introduction)

문서의 목적, 범위, 정의, 약어, 참조 문서, 문서 개요를 기술한다. 이해관계자가 문서를 처음 접할 때 전체 맥락을 파악할 수 있도록 한다.

- Purpose: 이 SRS가 무엇을 위한 것인지
- Scope: 시스템이 다루는 영역과 다루지 않는 영역
- Definitions/Acronyms: 도메인 용어 정의 (예: 콜드체인, HACCP, SLA)

### 섹션 2: 전체 설명 (Overall Description)

시스템의 맥락과 제약을 설명한다. 시스템이 어떤 환경에서 동작하는지, 어떤 외부 요소와 상호작용하는지 기술한다.

- Product Perspective: 시스템이 속한 더 큰 시스템과의 관계
- Product Functions: 시스템의 주요 기능 목록 (FR 요약)
- User Classes & Characteristics: 사용자 유형별 특성
- Operating Environment: 하드웨어, 소프트웨어, 네트워크 환경
- Design & Implementation Constraints: 기술 스택, 예산, 규제 제약
- Assumptions & Dependencies: 전제 조건 및 외부 의존성

### 섹션 3: 이해관계자 요구사항 (Stakeholder Requirements — StRS)

각 이해관계자가 시스템에 기대하는 것을 정리한다. PRD의 이해관계자 목록을 기반으로 각자의 관심사를 요구사항 형태로 표현한다.

### 섹션 4: 시스템 요구사항 (System Requirements — SyRS)

시스템이 반드시 충족해야 할 기능·비기능 요구사항을 ID 기반으로 열거한다. 각 요구사항에는 검증 가능한 수용 기준(Acceptance Criteria)이 포함되어야 한다.

- 기능 요구사항 (Functional): SRS-F-XXX 형태
- 비기능 요구사항 (Non-Functional): SRS-NF-XXX 형태

### 섹션 5: 인터페이스 요구사항 (Interface Requirements)

외부 시스템, 하드웨어, 사용자 인터페이스와의 연동 방식을 정의한다.

- 사용자 인터페이스 (UI)
- 하드웨어 인터페이스 (IoT 센서 등)
- 소프트웨어 인터페이스 (외부 API 연동)
- 통신 인터페이스 (프로토콜, 포맷)

### 섹션 6: 검증 (Verification)

각 요구사항이 충족되었음을 어떻게 확인할지 정의한다. 검증 방법(테스트, 인스펙션, 시연, 분석)과 합격 기준을 매핑한다.

### 섹션 7: 부록 (Appendices)

분석 모델, 이슈 목록, 용어 사전, 참조 데이터 등 본문에 포함하기 어려운 보충 자료를 첨부한다.

## 3. PRD → SRS 매핑 계획표

### 3.1 전체 섹션 매핑

| PRD 항목 | SRS 섹션 번호 | 변환 방식 | 추가 필요 사항 |
|----------|--------------|-----------|----------------|
| 1.1 목적 및 범위 | 1 (Introduction) | 비즈니스 목적을 시스템 범위로 재표현 | 시스템 경계 명확화 (In/Out Scope) |
| 1.2 이해관계자 목록 | 2, 3 (Overall Desc + StRS) | 역할별 요구사항으로 분해 | 이해관계자별 우선순위·권한 정의 |
| 2.1 핵심 문제 | 2 (Overall Desc) | 문제를 시스템 컨텍스트 다이어그램으로 시각화 | 외부 엔티티 식별 |
| 2.2 제품 목표 (KPI) | 4 (SyRS — NF) | 수치 목표를 NFR 수용 기준으로 변환 | 측정 주기·방법 추가 |
| FR-001 HACCP | 4 (SyRS — F), 5 (Interface) | 기능 동작 + 서류 인터페이스 명세 | 인증서 파일 포맷, 로트번호 체계 |
| FR-002 납기 SLA | 4 (SyRS — F, NF) | SLA 계약 로직 + 지연 보상 산정 알고리즘 | 보상 금액 계산식, 예외 케이스 |
| FR-003 콜드체인 | 4 (SyRS — F), 5 (Interface) | 온도 이탈 감지 로직 + IoT 인터페이스 | MQTT 프로토콜 명세, 알림 임계값 |
| FR-004 물류 추적 | 4 (SyRS — F), 5 (Interface) | 추적 링크 생성 로직 + 외부 물류 API | API 응답 포맷, 폴백 처리 |
| FR-005 통관 서류 | 4 (SyRS — F), 5 (Interface) | 서류 체크리스트 자동화 + EDI 연동 | UNI-PASS EDI 포맷, 서류 상태 추적 |
| FR-006 CS 창구 | 4 (SyRS — F), 5 (Interface) | 메시지 수신·발송 로직 + LINE API | 응답 SLA 타이머, 에스컬레이션 룰 |
| FR-007 교환 보증 | 4 (SyRS — F) | 클레임 접수·처리 워크플로우 | 선도 등급 판정 기준, 처리 상태 FSM |
| FR-008 샘플 | 4 (SyRS — F) | 샘플 발송 워크플로우 + 피드백 수집 | 피드백 폼 데이터 구조 |
| FR-009 품질 성적서 | 4 (SyRS — F), 7 (Appendix) | 성적서 발행 로직 + 데이터 스키마 | 중량 편차 계산 로직 |
| FR-010 이력 관리 | 4 (SyRS — F), 7 (Appendix) | CRUD + 월간 리포트 생성 | DB 스키마 (Order, Shipment, Claim) |
| 4.x 기술 스택 | 2 (Overall Desc — Constraints) | 제약 조건으로 SRS에 흡수 | 버전 명시, 라이선스 확인 |
| 5.x NFR | 4 (SyRS — NF) | NFR ID 부여 + 수용 기준 구체화 | 측정 도구 명시 |
| 6. DoD | 6 (Verification) | 완료 기준을 V&V 매트릭스로 변환 | 테스트 방법 구체화 |

## 4. SRS 작성 시 주의사항 (AI 에이전트 기반 개발 맥락)

### 4.1 측정 가능성 원칙

AI 에이전트가 요구사항을 구현 태스크로 분해하려면, 모든 수용 기준이 테스트 가능해야 한다. 모호한 표현은 에이전트가 잘못된 방향으로 구현할 수 있다.

- 나쁜 예: "빠르게 응답해야 한다"
- 좋은 예: "API 응답 시간 P95 기준 2초 이내"

### 4.2 ID 기반 추적성 확보

에이전트는 요구사항 ID를 통해 PRD ↔ SRS ↔ TASK ↔ 테스트를 연결한다. FR-001이 SRS-F-001, SRS-F-002로 분해되고, 각각이 TASK-C-001, TASK-L-001로 이어지는 체인이 완성되어야 한다.

### 4.3 예외 처리 명세 필수

AI 에이전트는 Happy Path만 구현하는 경향이 있다. SRS에 반드시 에러 케이스와 엣지 케이스를 명시해야 한다.

- 물류 API 타임아웃 시 동작
- IoT 센서 연결 끊김 시 알림 정책
- 통관 EDI 수신 거부 시 재시도 횟수

### 4.4 외부 의존성 명시

이 프로젝트는 통관 EDI(UNI-PASS), 물류 API(DHL/FedEx/EMS), LINE Messaging API, AWS IoT Core 등 외부 시스템 의존성이 높다. 각 외부 시스템의 가용성 SLA가 내부 SRS NFR에 영향을 미치므로 반드시 섹션 5에 명기하고, 폴백 처리를 SRS-F에 포함해야 한다.

### 4.5 규제 요구사항 우선 처리

HACCP, 일본 식품위생법, 수출신고 5년 보관 등 규제 요구사항은 비즈니스 의사결정보다 상위 제약이다. SRS에서 NFR-C(규정준수) 항목은 협상 불가(Non-negotiable)로 표시하고, AI 에이전트가 MVP 범위 조정 시 제외하지 않도록 명시해야 한다.

### 4.6 CQRS 분리 예고

이 서비스는 발주 생성(Command)과 납품 이력 조회(Query) 패턴이 뚜렷하게 분리된다. SRS 작성 시 각 기능이 Command인지 Query인지 태그를 달아두면, day3 TASK 추출 단계에서 CQRS 분리를 자연스럽게 적용할 수 있다.
